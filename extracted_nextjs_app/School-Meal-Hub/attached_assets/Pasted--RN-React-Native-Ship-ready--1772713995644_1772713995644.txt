당신은 프로덕션 경험이 매우 풍부한 “시니어 풀스택 + 시니어 RN(React Native) 마이그레이션 엔지니어”입니다.
목표는 “현재 개발 완료(또는 Ship-ready) 상태인 기존 웹앱(React/Next.js)”을 “React Native + Expo 기반 iOS/Android 네이티브 앱”으로 변환하는 것입니다.

단, 아래의 ‘절대 보존 규칙’은 단 하나도 예외 없이 지켜야 합니다.
오직 바뀔 수 있는 것은 런타임 환경뿐입니다: Web → Native (iOS/Android).
그 외 디자인/UX/구조/로직/데이터/흐름은 100% 동일해야 합니다.

────────────────────────────────────────────────────────
0) 절대 보존 규칙 (Non‑Negotiable)
────────────────────────────────────────────────────────
[UI/디자인]
- 색상, 폰트, 타이포그래피, 라인 높이, 자간, 여백, 패딩/마진, 모서리, 테두리, 그림자, 투명도, 아이콘 크기/위치, 컴포넌트 형태, 상태 UI(loading/empty/error), 애니메이션, 전환 효과까지
  “픽셀 단위로 동일”하게 복제한다.
- hover/focus/active 등 상태가 웹에 존재한다면 모바일에서도 “동일한 시각적 피드백”이 느껴지도록 그대로 재현한다
  (hover는 모바일에서 press/long-press/포인터 입력 환경까지 포함해 동일한 색/강도로 매칭).

[레이아웃/구조]
- 화면 구성, 페이지 구조, 컴포넌트 계층, UI 섹션 순서, 화면 전개 방식(플로우)은 100% 동일해야 한다.
- “모바일 최적화” “네이티브스럽게 재해석” “재디자인” “정보 재배치” “바텀탭 추가” 같은 변경은 절대 금지.
  (웹에 이미 있던 내비게이션 패턴만 그대로 복제)

[UX/동작]
- 클릭/탭/스크롤/폼 입력/검증/모달/토스트/전환/리다이렉트/상태 유지 등
  사용자가 느끼는 경험이 웹과 “완전히 동일”해야 한다.
- 제스처/트랜지션 또한 웹의 동작을 그대로 모사한다(새로운 네이티브 패턴 금지).

[로직/아키텍처]
- 상태관리, 비즈니스 로직, 데이터 흐름, API 연동, 프로세스, 정책/규칙 엔진, 검증(Zod 등),
  “코어 로직”은 단 1줄도 의미가 바뀌면 안 된다.
- 리팩토링/재설계 금지. 단, “플랫폼 차이로 인해 동일 동작을 만들기 위한 최소 래핑/어댑터”만 허용한다.
- 기존 AI 코칭/AI 채팅 기능이 없다면 절대 추가하지 않는다(새 기능 금지).

[정리]
- 결과물은 “기존 웹앱이 그대로 iOS/Android 네이티브로 실행되는 것처럼”
  시각적/기능적으로 구분 불가능해야 한다.

────────────────────────────────────────────────────────
1) 기술 스택(고정) & 전환 전략(고정)
────────────────────────────────────────────────────────
- 반드시 React Native + Expo로 구현한다.
- 웹앱 백엔드(Next.js API/서버)는 “그대로 유지”한다. (서버를 RN로 옮기거나 재작성 금지)
- 모바일 앱은 기존 서버의 동일 API를 호출하여 완전히 동일한 데이터/세션/정책으로 동작해야 한다.

[라우팅/구조 보존]
- 웹의 라우트/페이지 구조를 최대한 1:1로 보존하기 위해 Expo Router를 사용한다.
- 기존 Next.js의 route/page 개념과 동일하게 매핑한다:
  - 웹 route: /dashboard → 모바일 route: /dashboard (동일 경로명/동일 화면명)
  - 웹 route: /admin/... → 모바일에서도 동일한 path로 존재(권한 정책 동일)

[스타일/디자인 토큰 보존]
- 웹이 Tailwind 기반이라면 RN에서도 NativeWind(또는 동등한 Tailwind-to-RN 솔루션)로
  “동일한 tailwind config / 동일 토큰”을 사용해 픽셀 수준의 스타일 재현을 우선한다.
- 폰트는 웹과 동일 폰트 파일을 Expo(Font)로 로드해 동일한 타이포그래피를 재현한다.

[상태관리/로직 공유]
- 가능하면 기존 TypeScript 로직(정책 엔진/유틸/검증/타입/상수)을 그대로 재사용한다.
- monorepo 권장 구조:
  /apps/web (기존 Next)
  /apps/mobile (Expo)
  /packages/shared (types, zod schemas, policy engines, api client, constants, date utils)
- “로직 변경”이 아니라 “로직 위치를 공유 패키지로 이동”하는 것은 허용(단 의미 변경/리팩토링 금지).

────────────────────────────────────────────────────────
2) 플랫폼 차이 허용 범위(딱 이것만 허용)
────────────────────────────────────────────────────────
- HTML 태그를 RN 컴포넌트로 치환하는 “렌더링 레이어” 변경만 허용:
  div/span → View/Text
  button → Pressable
  input/select → TextInput/Picker(단, 웹과 동일한 UI 느낌으로 커스텀 렌더)
  a/link → expo-router Link
- 단, 치환 후에도 화면은 픽셀/동작이 동일해야 한다.
- 성능 최적화는 “동작/외관이 바뀌지 않는 범위”에서만 가능.

────────────────────────────────────────────────────────
3) 세션/인증(가장 중요) — 웹과 100% 동일하게 유지
────────────────────────────────────────────────────────
- 웹이 세션 쿠키 기반이면, 모바일도 세션 쿠키 기반을 그대로 유지해야 한다(토큰 방식으로 변경 금지).
- RN/Expo 환경에서 쿠키 유지가 어려우면:
  - 쿠키 저장/전송을 “동일하게” 보장하는 라이브러리/어댑터를 도입하되,
  - 이것은 ‘런타임 차이 보정’으로만 인정된다(서버 인증 방식 변경 금지).
- Expo Go 제약으로 불가능한 경우:
  - “커스텀 Dev Client / EAS Build”를 사용해 동일 세션 동작을 보장한다.
- 모든 API 호출은 기존과 동일한 헤더/쿠키/credentials 정책을 유지해야 한다.

────────────────────────────────────────────────────────
4) 테스트/검증 — ‘동일성’이 합격 기준
────────────────────────────────────────────────────────
다음 검증을 통과해야 “전환 완료”로 간주한다.

[시각적 동일성]
- 핵심 화면들에 대해 스크린샷 비교(시각 회귀 테스트)로 픽셀 차이를 검증한다.
- 최소 대상: 로그인/회원가입/대시보드/혼잡/예약/스토어/알림/설정/관리자 콘솔 핵심 화면

[플로우 동일성]
- E2E 시나리오로 웹과 동일 플로우를 검증:
  1) 로그인 → 대시보드 → 혼잡 확인 → 예약 → 알림 확인
  2) 포인트 스토어 진입 → 교환 → 내역 확인
  3) 운영자 콘솔(권한) → 상품 등록/재고 변경 → 교환 처리
  4) 오프라인/NEIS 실패/워커 지연 상황에서 fallback이 웹과 동일한 UX로 유지

[로직 동일성]
- shared 로직 모듈에 대해 단위 테스트로 결과 동일성을 검증한다(입력→출력 동일).

────────────────────────────────────────────────────────
5) 산출물(반드시 제공해야 함)
────────────────────────────────────────────────────────
- “전체 코드베이스 전환 결과”를 제공하라. 부분 샘플 금지.
- 반드시 포함:
  1) 최종 폴더/파일 트리(모노레포 포함)
  2) mobile(Expo) 앱 전체 코드 + 실행 방법
  3) shared 패키지 구성(있다면) + 어떤 로직이 공유되는지 명시
  4) 환경변수(.env.example) 정리
  5) iOS/Android 빌드/실행 가이드(로컬 + EAS)
  6) 동일성 검증(스크린샷/플로우 테스트) 설정 및 실행 가이드

────────────────────────────────────────────────────────
6) 작업 절차(너는 이 순서대로 실행한다)
────────────────────────────────────────────────────────
1) 기존 웹앱 분석:
   - 라우트 목록, 공통 레이아웃, 컴포넌트 라이브러리, 스타일 시스템(Tailwind/토큰),
     상태관리, API 클라이언트, 인증(세션/쿠키), 워커/알림 구조를 정리한다.
2) Expo 프로젝트 생성 및 구조 세팅:
   - expo-router로 웹 라우트 구조를 1:1 매핑한다.
3) 스타일/폰트/토큰 1:1 이식:
   - tailwind config/색/타이포를 그대로 재현한다.
4) 공통 로직 공유(가능하면):
   - zod 스키마, 정책 엔진, 상수/유틸을 packages/shared로 이동(의미 변경 금지).
5) 화면을 “동일하게” 구현:
   - 웹 화면 1개씩 RN으로 치환하면서 픽셀/동작 동일성 체크를 반복한다.
6) 세션/쿠키 동작 완성:
   - 로그인/세션 유지/로그아웃이 웹과 완전히 동일하게 동작하도록 보장한다.
7) 오프라인/에러/fallback 동일성 보정:
   - 웹의 로딩/빈상태/에러 UI를 RN에서도 동일하게 구현한다.
8) 테스트 자동화:
   - 스크린샷 회귀 + E2E + 단위 테스트를 구성한다.
9) 최종 점검 체크리스트 제출:
   - “달라진 점: 0개”를 목표로, 불가피한 차이가 있다면 ‘기술적 한계’로 명확히 목록화하고
     그 차이도 시각/동작적으로 동일하게 느껴지도록 우회 구현한다.

────────────────────────────────────────────────────────
7) 너의 출력 방식(중요)
────────────────────────────────────────────────────────
- 먼저 “전환 설계 요약(1페이지)”를 제시한다(아주 간단히).
- 그 다음 바로 코드/파일 트리/실행 방법/테스트 방법을 제공한다.
- 절대 “모바일에 맞게 바꾸자”는 제안을 하지 말고, 오직 ‘동일하게 복제’만 수행한다.

이제 위 요구사항을 만족하도록, 현재 웹앱 전체를 React Native + Expo 네이티브 앱으로 100% 동일하게 전환하는 작업을 시작하라.
