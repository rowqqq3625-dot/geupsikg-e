당신은 프로덕션 경험이 풍부한 시니어 풀스택 엔지니어입니다.
우리는 급식E(Next.js App Router + TS + Tailwind + Prisma + Postgres)를 개발했고,
프로토타입 기능은 이미 구현되어 있습니다:

- 학생: NEIS 급식, 혼잡 상태/추천 입장, 예약(가상 줄서기), 체크인, 습관 기록, 목표(선택), 주간 리포트(규칙 기반), 배지, 포인트, 포인트 스토어 교환
- 운영자(SCHOOL_ADMIN): 혼잡 설정/히트맵(집계), 스토어 상품/교환 처리, 신고/모더레이션, 참여 지표(집계)
- 기관 도입: /org/signup 신청 → SUPER_ADMIN 승인 → 초대 토큰으로 운영자 생성
- 알림: 인앱 알림(필수) + 푸시(옵션), 워커 프로세스 운영
- AI 코칭/AI 채팅/LLM 관련 기능은 “없어야” 합니다(절대 추가하지 말 것).

이번 작업은 “프로토타입 마무리(Ship-ready)” 단계입니다.
새 기능을 추가하지 말고, 아래 품질/UX/운영/보안 보정만 수행해
학교 파일럿에서 실제로 끊김 없이 돌아가는 완성형 프로토타입으로 마감하세요.

=====================================================
[절대 규칙] 스코프 고정
=====================================================
- 신규 핵심 기능 금지(새 DB 도메인/새 탭/새 서비스 추가 금지).
- 기존 플로우의 누락/버그/엣지케이스/성능/UX 일관성만 개선.
- “애플처럼” 미니멀: 흰 배경 + gray-50 섹션, 얇은 border, 절제된 blue, 과한 애니메이션/그라데이션/그림자 금지.

=====================================================
1) 최종 합격 기준(ACCEPTANCE CRITERIA)
=====================================================
A. 홈(대시보드) 1스크린 완성
- 첫 화면에 핵심 4개만: (1)오늘 급식 (2)혼잡/추천입장+예약 (3)포인트+스토어 (4)바로가기(인증/피드백/매칭 중 1~2개)
- 나머지는 탭/더보기로 이동. 정보 과부하 금지.

B. 모든 화면 상태 UI 통일
- 로딩: Skeleton
- 빈 상태: EmptyState(한 줄 + 버튼 1개)
- 에러: ErrorState(재시도 + requestId 작게)
- 성공: Toast(짧고 절제)

C. 트랜잭션/중복 방지 무결성 “완벽”
- 스토어 교환: 포인트 차감 + 장부 기록 + 재고 감소 + 교환 생성이 1 트랜잭션(원자성).
- 교환 취소 환불: 중복 환불 방지(예: refundedAt 또는 idempotency)까지 포함.
- 예약 생성: 1인 1예약, 슬롯 정원 제한 트랜잭션.
- 체크인/채팅/교환/예약/업로드: 레이트리밋 확실히 적용.

D. 권한/스코프 “완벽”
- STUDENT / SCHOOL_ADMIN / SUPER_ADMIN 권한 가드 철저.
- 운영자 API는 기본 “자기 schoolId만”.
- SUPER_ADMIN만 기관 신청 승인 가능.
- 관리자 UI에서 개인 raw 체크인/개인 리포트/개인 민감 데이터 노출 금지(집계 중심).

E. 워커/스케줄 신뢰성
- worker가 죽어도 웹이 깨지지 않게: 캐시/마지막 결과 표시.
- 워커 재기동 시 중복 알림/중복 집계가 생기지 않게 idempotent upsert로 구현.
- “개발용 tick”이 있다면 유지하되, 프로덕션에서는 worker가 주체.

F. 오프라인/저품질 네트워크 UX
- 오프라인이면 마지막으로 본 “급식/혼잡/포인트” 요약을 표시하고, 온라인 복귀 시 자동 갱신.
- 오프라인 배지 표시(절제), 기능 버튼은 비활성/안내.

G. 기관 도입 신청 UX/운영
- /org/signup 제출 후 상태 메시지/페이지 일관성.
- SUPER_ADMIN 승인 → 초대 토큰 → 운영자 생성/로그인 흐름 끊김 없이 동작.
- AuditLog에 승인/초대/운영자 생성 주요 이벤트 기록.

=====================================================
2) 반드시 수행할 “보정 작업” 목록(구체)
=====================================================
(1) UI/UX 정리
- 공통 컴포넌트: Card/Button/Input/Modal/Skeleton/Toast/EmptyState/ErrorBoundary 통일
- 네비게이션 정리: 학생 탭(홈/혼잡/스토어/내정보 또는 홈/더보기)로 단순화
- 카피(문구) 다듬기: 짧고 명확(“AI”라는 단어 사용 금지)

(2) API 표준화/에러 표준화
- 성공: { ok:true, data }
- 실패: { ok:false, error:{ code, message, requestId } }
- requestId를 서버에서 생성/전파(로그/응답/클라 표시)

(3) 성능/DB 최적화
- 자주 호출되는 API 기준으로 쿼리 최적화 + 필요한 인덱스 추가/검증:
  - /api/crowd/status, /api/crowd/recommendation
  - /api/store/items, /api/store/redeem
  - /api/notifications, /api/admin/overview
- 불필요한 N+1 제거, 필요한 필드만 select

(4) 관측/운영 로그
- AuditLog: 운영자 액션(상품 수정/교환 처리/혼잡 설정/신고 조치/기관 승인) 기록
- 서버 로그 구조화: requestId, role, schoolId 정도(개인정보 최소)
- 에러 추적은 옵션(키 있으면 연동, 없으면 최소 로깅)

(5) 테스트(프로토타입 마감에 필수)
- 단위 테스트(최소):
  - 포인트 스토어 교환/환불 트랜잭션 idempotency
  - 예약 생성 정원/1인1예약 제약
  - 권한 가드(authz)
- E2E(추천: Playwright) 최소 시나리오:
  1) 학생 가입→홈→혼잡 확인→예약→스토어 교환
  2) 운영자 초대 생성→/admin에서 상품 등록→교환 처리
  3) 기관 신청→SUPER_ADMIN 승인→초대 토큰→운영자 생성

(6) 배포/실행 절차 최종 정리
- .env.example 정리(필수/옵션 구분)
- docker-compose(최소 postgres, 필요 시 minio 유지)
- scripts:
  - npm run dev
  - npm run worker
  - prisma migrate dev / deploy 구분
  - seed(데모용 학교/상품/유저) 옵션
