당신은 프로덕션 경험이 풍부한 시니어 풀스택 엔지니어입니다.
우리는 급식E를 Next.js(App Router)+TypeScript+Tailwind+Prisma+Postgres로 개발 중이며,
1~5단계(세션 Auth/보호, 학교코드+NEIS 급식, 피드백→포인트/랭킹, 클린플레이트, 알림/워커, Food Buddy 매칭/모더레이션)까지 동작하는 상태입니다.

이제 “6~8단계 통합 최종 구현”을 설계하고 실제 코드를 수정/추가하여 완성하세요.
이번 통합 단계의 핵심은:
1) 혼잡 예측/추천 입장/가상 줄서기(예약) 엔진을 완전 구동
2) 습관 트래킹 + 건강 목표(선택) + 주간 리포트(규칙 기반, AI/채팅 없음) 완성
3) 포인트 스토어(학교 지정 상품 교환) 완성
4) 기관(학교 운영 측) 전용 도입 신청(회원가입) 페이지/흐름 완성
5) 실서비스 수준 UX/성능/보안/운영(관리자 콘솔, 관측, 레이트리밋, 오프라인/캐시)까지 마무리

중요: “AI 코칭”과 “AI 채팅” 기능은 완전히 삭제한다.
- LLM 연동, AI 문구 생성, 코치/챗 UI/API/DB 테이블은 만들지 않는다.
- 대신, 개인화는 ‘규칙 기반 정책 엔진(Policy Engine)’로만 구현한다.
- 문구는 템플릿/룰 기반으로 짧게 생성한다(“AI” 표현 금지).

0) 제품 원칙(반드시 코드/UX/정책에 반영)
=====================================================
A) 입력 최소, 개인화 최대(Δ2)
- 학생 가입 입력은 필수 최소(학교/학년/반/학번) 유지.
- 습관/목표는 “선택 입력”, 미입력이어도 기본 기능(급식/혼잡/예약/포인트/스토어)은 완전 동작.

B) 학생 친화적(Apple-like 미니멀)
- 흰 배경 + gray-50 섹션, 얇은 border, 절제된 blue 포인트
- 과한 그림자/그라데이션/애니메이션 금지
- 대시보드는 1스크린에 핵심 카드 4개만(급식/혼잡/포인트/바로가기)

C) 실패해도 깨지지 않음
- NEIS 실패/데이터 부족/워커 지연/푸시 미허용/오프라인에서도 UI가 무너지지 않게
  “캐시된 마지막 데이터 + 대체 문구 + 재시도 버튼” 제공.

D) 프라이버시/안전
- 혼잡은 익명 집계만 표시(개인 체크인 raw를 관리자/사용자에게 노출 금지).
- Food Buddy는 신고/차단/정지/모더레이션 유지.
- 관리자 화면은 학교 단위 집계 중심(개인 데이터 최소).

E) 운영 가능성(학교 파일럿 기준)
- 학교 운영자는 “혼잡 설정/스토어 상품/신고/승인/지표”를 한 곳에서 관리 가능해야 함.
- 감사 로그(AuditLog)로 관리자 행동 추적.

1) 이번 통합 단계에서 반드시 완성할 기능(정의)
=====================================================
[혼잡/예약 엔진]
1) 체크인 이벤트(INTENT/ENTER/EXIT) 수집 + 레이트리밋
2) 5분 버킷 스냅샷(CrowdSnapshot) 집계(워커)
3) 예측 캐시(WaitTimePrediction) 생성(워커, 규칙/통계 기반)
4) 추천 입장 시간(다음 30분 최적 버킷) 제공
5) 가상 줄서기/입장 예약(Time Slot) + 슬롯 정원 제한 + 예약 임박 알림(인앱 필수, 푸시 옵션)

[습관/목표/리포트(규칙 기반, AI 없음)]
6) 습관 트래킹(10초 체크): 아침 여부, 물, 활동, 기분(선택)
7) 건강 목표(선택): MAINTAIN/BALANCE/ENERGY/GAIN/LOSE (안전 톤)
8) 주간 리포트(워커): 지난 주 요약 + 다음 주 1개 포커스(규칙 기반)
9) 배지(최소 세트) + 지급 알림(인앱)

[포인트 스토어]
10) 학교 운영자가 등록한 상품(쿠폰/매점/자판기/특권 등)을 학생이 포인트로 교환
11) 재고/수량/교환 제한/중복 방지/포인트 차감은 트랜잭션으로 원자적 처리
12) 교환 내역/상태(REQUESTED/APPROVED/READY/COMPLETED/CANCELLED) + 운영자 처리 UI

[기관 전용 도입 신청(학교 운영 측)]
13) “기관 전용 회원가입(도입 신청)” 페이지(/org/signup):
    - 학교 관계자(교사/행정실/영양사/교감/교장 등)가 학교 서비스를 신청
    - 신청만으로 학생 DB가 열리면 안 됨(승인/초대 기반)
14) 승인 플로우:
    - (MVP) 시스템 슈퍼관리자(SUPER_ADMIN) 승인 후, 해당 학교의 ADMIN 계정 생성/초대 링크 발급
    - 승인 전에는 운영 콘솔 접근 불가
15) 기관 운영 콘솔(/admin):
    - 혼잡 설정/히트맵(집계)
    - 포인트 스토어 상품/교환 요청 처리
    - 신고/모더레이션(기존 기능 연동)
    - 참여 지표(집계): 체크인 수, 예약 수, 피드백 수, 클린플레이트 승인 수, 스토어 교환 수

[실서비스 품질/운영/보안]
16) 공통 로딩/에러/빈 상태 UI 통일(Skeleton/Toast/EmptyState/ErrorBoundary)
17) 캐시/재검증 전략: 급식/혼잡/리포트는 DB 캐시 우선 + stale 표시
18) 레이트리밋/권한 가드/스코프 강제(schoolId 기반)
19) 감사 로그(AuditLog) 기록
20) PWA 최소: 설치 가능 + 오프라인에서 “마지막 데이터” 표시

2) 기술 스택/구현 방식(선택 고정)
=====================================================
- Next.js App Router + TS + Tailwind + Prisma + Postgres 유지
- 알림: 인앱(Notification 테이블) 필수, 푸시(Web Push)는 옵션(4단계 구성 재사용 가능)

3) 데이터 모델(Prisma) — 최종 통합 스키마(추가/정리)
=====================================================
기존(User, School, PointLedger, Notification, UserEvent, MealCache, CleanPlateSubmission, Buddy*, Moderation*)는 유지.

(권한/기관)
A) User.role 확장:
- enum: STUDENT | SCHOOL_ADMIN | SUPER_ADMIN
- 기존 ADMIN이 있다면 SCHOOL_ADMIN으로 정리(마이그레이션 포함)
- User는 schoolId 소속(학생/학교관리자), SUPER_ADMIN은 schoolId nullable

B) OrganizationApplication (기관 도입 신청)
- id (uuid)
- schoolName (string)
- contactName (string)
- contactRole (string)            // “교사/행정실/영양사/교감/교장”
- contactEmail (string)
- contactPhone (string nullable)
- note (string nullable, max 300)
- status enum: PENDING | APPROVED | REJECTED
- decidedByUserId (uuid nullable, relation User)
- decidedAt (datetime nullable)
- createdAt, updatedAt
index: (status, createdAt)

C) AdminInviteToken (승인 후 초대 링크)
- id (uuid)
- schoolId (relation School)
- token (string unique)
- expiresAt (datetime)
- usedAt (datetime nullable)
- createdAt

(혼잡/예약: 6단계)
D) CafeteriaCheckinEvent
E) CrowdSnapshot
F) WaitTimePrediction
G) QueueReservation
H) CafeteriaConfig
(위 6단계 스키마 그대로, 집계/유니크/인덱스 포함)

(습관/목표/리포트: 7단계에서 AI 제거 버전)
I) HealthGoal (선택)
- userId unique
- goalType enum: MAINTAIN | BALANCE | ENERGY | GAIN | LOSE
- weeklyFocus enum: PROTEIN | VEGGIE | WATER | BREAKFAST | SLOW_EAT | NONE (nullable)
- motivation(optional, max 80)
- createdAt, updatedAt

J) HabitDailyLog
- (userId, date) unique
- breakfast YES/NO/UNKNOWN
- waterCups/activityMinutes/mood/notes(선택)
- createdAt, updatedAt

K) WeeklyReport (규칙 기반)
- (userId, weekStart) unique
- summary(string)
- highlights(Json)
- nextWeekPlan(Json)
- createdAt

L) BadgeDefinition / UserBadge (없으면 추가)
- 최소 배지 세트는 유지(스트릭/클린플레이트/피드백/혼잡/매칭 등)

(포인트 스토어)
M) StoreItem (학교별 상품 카탈로그)
- id (uuid)
- schoolId (relation)
- name (string)
- description (string nullable, max 300)
- imageUrl (string nullable)      // 외부 업로드는 이후, 지금은 optional
- pricePoints (int)              // 교환 포인트
- stock (int nullable)           // null이면 무제한
- perUserDailyLimit (int nullable)   // 선택
- isActive (boolean default true)
- createdAt, updatedAt
index: (schoolId, isActive)

N) Redemption (교환 요청/처리)
- id (uuid)
- schoolId (relation)
- userId (relation)
- itemId (relation StoreItem)
- quantity (int default 1)
- pointsSpent (int)
- status enum: REQUESTED | APPROVED | READY | COMPLETED | CANCELLED
- adminNote (string nullable)
- processedByUserId (uuid nullable, relation User)
- processedAt (datetime nullable)
- createdAt, updatedAt
index: (schoolId, status, createdAt)
유의: 포인트 차감/재고 감소는 트랜잭션으로 원자적 처리

(운영/감사)
O) AuditLog
- id, schoolId(nullable), actorUserId(nullable), action, targetType, targetId, meta(Json minimal), createdAt
indexes: (schoolId, createdAt), (actorUserId, createdAt)

5) 워커(백그라운드 잡) — 최종 통합 스케줄
=====================================================
- npm run worker로 실행되는 워커를 유지/확장.

B) 주간 리포트/배지
- 매주 월요일 07:30: WeeklyReport 생성(규칙 기반)
- 매일 1회: Badge 룰 평가 → UserBadge 지급 + 알림

C) 캐시/워밍(선택)
- NEIS 급식 캐시 프리워밍(이미 있으면 유지)

6) API 라우트(백엔드) — 필수 목록
=====================================================
[기관 도입 신청]
1) GET /org/signup (page) + POST /api/org/apply
- body: { schoolName, contactName, contactRole, contactEmail, contactPhone?, note? }
- 결과: { ok:true, applicationId }

2) (SUPER_ADMIN) GET /api/super/org-applications?status=PENDING
3) (SUPER_ADMIN) POST /api/super/org-applications/{id}/decide
- body: { action:"APPROVE"|"REJECT" }
- APPROVE 시:
  - School 생성(또는 매핑)
  - AdminInviteToken 발급
  - AuditLog 기록

4) GET /org/invite/{token} (page)
- 토큰 유효하면 “학교 운영자 계정 생성/로그인” 진행(세션 발급)
- 생성되는 유저 role=SCHOOL_ADMIN, schoolId 지정

[혼잡/예약]
5) POST /api/crowd/checkin
6) GET /api/crowd/status
7) GET /api/crowd/recommendation
8) POST /api/queue/reserve
9) POST /api/queue/cancel
10) GET /api/queue/me

[습관/목표/리포트]
11) POST /api/habits/today (upsert)
12) GET /api/habits/history?days=14
13) GET/PUT /api/goal
14) GET /api/reports/weekly?cursor=
15) GET /api/badges/me

[포인트 스토어]
16) GET /api/store/items
- 학생용: 내 schoolId의 활성 상품 리스트

17) POST /api/store/redeem
- body: { itemId, quantity }
- 트랜잭션 필수:
  - User.points 충분 여부
  - 재고(stock) 충분 여부
  - 제한(perUserDailyLimit 등) 확인
  - Redemption 생성
  - PointLedger에 -points 기록(reason="STORE_REDEEM", refId=redemptionId)
  - User.points 감소
  - 재고 감소(재고가 null이면 생략)
- 응답: { ok:true, redemptionId, newPoints }

18) GET /api/store/me/redemptions
- 내 교환 내역

(학교 운영자)
19) (SCHOOL_ADMIN) POST /api/admin/store/items (create)
20) (SCHOOL_ADMIN) PUT /api/admin/store/items/{id} (update/activate/deactivate/stock)
21) (SCHOOL_ADMIN) GET /api/admin/store/redemptions?status=REQUESTED
22) (SCHOOL_ADMIN) POST /api/admin/store/redemptions/{id}/status
- body: { status:"APPROVED"|"READY"|"COMPLETED"|"CANCELLED", note? }
- CANCELLED이면 포인트 환불 정책:
  - (권장) 환불 수행: PointLedger에 +points(reason="STORE_REFUND", refId=redemptionId), User.points 증가
  - 트랜잭션 + 중복 환불 방지 필드(예: refundedAt nullable) 추가 가능

[운영 콘솔 통합]
23) (SCHOOL_ADMIN) GET /api/admin/overview
- 집계 지표만:
  - 오늘 체크인 수, 예약 수, 피드백 수, 클린플레이트 승인 수, 교환 요청 수, 신고 OPEN 수

24) (SCHOOL_ADMIN) GET/PUT /api/admin/cafeteria-config
25) (SCHOOL_ADMIN) GET /api/admin/crowd/overview?date=...
(기존 신고/모더레이션/클린플레이트 승인 API는 유지/통합)

공통:
- 모든 입력은 Zod 검증
- 에러 형식 통일: { ok:false, error:{ code, message, requestId } }
- authz로 role + schoolId 스코프 강제
- rate-limit 유틸 적용(체크인/예약/채팅/교환)

C) /habits, /reports, /badges
- 습관 10초 체크, 14일 스트릭
- 주간 리포트 리스트
- 배지 그리드

D) /store (학생용 포인트 스토어)
- 상품 리스트(가격/재고/설명)
- 교환 버튼(확인 모달)
- 내 교환 내역(/store/history)

기관/운영자용
E) /org/signup (기관 도입 신청)
- 학교명/담당자/이메일/역할/메모 입력(학생 가입과 완전 분리)
- 신청 후 상태 화면(/org/status/{applicationId}) 또는 간단 메시지 제공

F) /org/invite/[token]
- 토큰으로 학교 운영자 계정 생성/로그인(역할 SCHOOL_ADMIN)

G) /admin (학교 운영 콘솔 홈)
- 카드형 대시보드:
  - 신고 OPEN
  - 클린플레이트 대기(있다면)
  - 혼잡 히트맵/설정
  - 스토어 교환 요청
  - 참여 지표(집계)
- 스토어 관리(/admin/store):
  - 상품 CRUD, 활성/비활성, 재고 수정
  - 교환 요청 처리(상태 변경/환불)

H) PWA 최소
- 설치 가능(과하지 않은 배너)
- 오프라인: 마지막 데이터 표시 + 온라인 복귀 시 자동 갱신

=====================================================
8) 품질/운영/보안(8단계 패키지로 반드시 포함)
=====================================================
- 공통 컴포넌트: Skeleton/Toast/EmptyState/ErrorBoundary 구현 및 전 페이지 적용
- 공통 API client: requestId 포함, 에러 표준화, 재시도 훅
- 캐시/재검증: 급식/혼잡/리포트는 DB 캐시 우선, “마지막 업데이트 시각” 표시
- 관측:
  - 구조화 로그(요청ID, schoolId 정도만)
  - AuditLog: 관리자 액션/승인/스토어 처리/혼잡 설정 변경 기록
- 권한:
  - SUPER_ADMIN: 기관 신청 승인, 시스템 레벨
  - SCHOOL_ADMIN: 자기 schoolId 범위 운영
  - STUDENT: 학생 기능
- 개인정보 최소화:
  - 혼잡은 집계만 반환
  - 운영 콘솔에서 개인 단위 원천 이벤트/개인 리포트 노출 금지
- 레이트리밋:
  - 체크인 60초, 예약 10초, 교환 10초, 채팅 10초 내 3회 등(정책 파일로 관리)

=====================================================
9) 출력 규칙(당신이 제공해야 할 산출물)
=====================================================
- 장문 설명 금지. 바로 구현 가능한 산출물 중심.
- 반드시 포함:
  1) 변경/추가 파일 트리
  2) prisma/schema.prisma 전체 + migrate 방법
  3) .env.example (필수/옵션 구분)
  4) 핵심 파일 “완전한 코드”(복붙 가능):
     - authz/rate-limit/requestId/에러 표준화 유틸
     - worker 업데이트(혼잡/예약 알림 + 주간리포트/배지)
     - 기관 신청(/org/signup) + 승인(SUPER_ADMIN) + 초대 토큰 플로우
     - 포인트 스토어(학생) + 운영자 스토어 관리(상품/교환 처리)
     - /dashboard 최종 UX 개편
     - /admin 통합 운영 콘솔
     - PWA 최소(서비스워커/오프라인 last-known-data)
  5) 로컬 실행/테스트 시나리오(짧게)

=====================================================
10) 로컬 테스트 시나리오(짧게)
=====================================================
1) 학생: 혼잡 체크인 → 예측/추천 표시 → 예약 → 임박 알림 생성 확인
2) 학생: 스토어 상품 교환 → 포인트 차감/장부 기록/재고 감소 트랜잭션 확인
3) 운영자: 상품 등록/재고 수정/교환 승인·완료·취소(환불) 확인 + AuditLog 확인
4) 기관: /org/signup 신청 → SUPER_ADMIN 승인 → 초대 토큰으로 SCHOOL_ADMIN 생성/로그인 확인
5) 오프라인/NEIS 실패 상황에서도 홈이 깨지지 않고 캐시/대체문구로 유지되는지 확인
6) 권한 스코프: 운영자는 자기 학교 데이터만, SUPER_ADMIN만 기관 승인 가능 확인

이제 위 요구사항을 만족하는 “급식E 6~8단계 최종 통합 구현”을 설계하고,
실제 코드(파일 트리 + 핵심 파일 전체 코드)를 출력하세요.
