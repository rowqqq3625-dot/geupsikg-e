당신은 프로덕션 경험이 풍부한 시니어 풀스택 엔지니어입니다.
프로젝트명: 급식E (학교 급식 코치 + 포인트 + 커뮤니티로 확장되는 SaaS)
지금은 “구현 1단계”로, 프론트엔드 + 백엔드 + DB까지 갖춘 강력한 기본 골격을 한 번에 세팅합니다.
목표는 “진짜 서비스처럼 로그인/회원가입/세션/DB저장/대시보드 보호”가 동작하는 MVP 기반입니다.

[필수 스택(결정)]
- Frontend/Backend 통합 풀스택: Next.js 14+ (App Router) + TypeScript
- Styling: Tailwind CSS (Apple-like 미니멀 UI)
- Backend API: Next.js Route Handlers (/app/api/...)
- Validation: Zod
- DB: PostgreSQL
- ORM: Prisma
- Auth: “세션 쿠키 기반” (JWT 또는 iron-session 등 라이브러리 1개 선택해 구현)
- Password는 이번 단계에서는 쓰지 않음.
  대신 “학교명 + 학년 + 반 + 학번”을 기반으로 계정 생성/로그인(개발용) 하되,
  추후 OTP/학교인증으로 바꿀 수 있도록 구조를 깔끔하게 분리해 둠.
- 개발 환경: docker-compose로 Postgres 실행 가능하게 구성

[중요 출력 규칙]
- 설명을 길게 하지 말고, “구현에 필요한 결과물”을 중심으로 출력.
- 반드시 아래를 포함해서 출력:
  1) 프로젝트 폴더/파일 트리
  2) 핵심 파일들의 전체 코드(복사-붙여넣기 가능한 수준)
  3) 로컬 실행 방법(명령어)
- 불필요한 장식/샘플 페이지 남발 금지. 필요한 페이지만.
- UI는 Apple처럼: 흰 배경, 옅은 구분선, 절제된 컬러, 여백과 타이포 중심.
  과한 그림자/그라데이션/형광색/애니메이션 금지.

[도메인 요구사항(이번 1단계 범위)]
A. 사용자 데이터(개인화의 최소 기반)
- 회원가입 시 입력: schoolName, grade(1~6), classNum(1~3), studentNumber, heightCm(optional), weightKg(optional), allergies(optional string[])
- 로그인 시 입력: schoolName, grade, classNum, studentNumber
- DB에 저장하고, 로그인 후에는 세션으로 “내 정보”를 조회 가능해야 함.

B. 페이지(프론트엔드)
- /login : 로그인 폼(학교명/학년/반/학번) + 회원가입 링크
- /signup : 회원가입 폼(로그인 필드 + 키/몸무게/알레르기 5개 체크박스) + 로그인 링크
- /dashboard : 로그인한 사용자만 접근 가능(미들웨어 또는 서버에서 보호)
  - 상단에 “OO학교 · 2학년 1반 · 13번” 형태로 표시
  - 카드 4개(전부 placeholder여도 됨):
    1) 오늘 급식(메뉴 placeholder + AI 분석 placeholder + 알레르기 배지 placeholder)
    2) 포인트(0P 고정) + 버튼 2개(클린플레이트 인증/건의·평점)
    3) 혼잡 예측(대기시간/추천입장 placeholder)
    4) Food Buddy(“적게/많이” 버튼 placeholder)
- 디자인 시스템:
  - 배경: white + section 구분은 gray-50
  - 텍스트: slate-900/700
  - 버튼 Primary: blue-600(Apple blue 계열 느낌), Secondary: border
  - 컴포넌트: rounded-2xl + border-slate-200 중심, shadow는 최소

C. API(백엔드)
- POST /api/auth/signup
  - body 검증(Zod)
  - 사용자 생성(중복이면 적절히 에러)
  - 세션 발급(쿠키 set)
- POST /api/auth/login
  - body 검증(Zod)
  - 사용자 조회(없으면 에러)
  - 세션 발급(쿠키 set)
- POST /api/auth/logout
  - 세션 쿠키 제거
- GET /api/me
  - 세션 기반으로 내 프로필 반환
- (선택) GET /api/meals/today
  - 아직 NEIS는 연결하지 말고, seed된 placeholder 오늘 급식 1건을 반환(추후 교체 가능하도록)

D. DB 스키마(Prisma)
- User
  - id (uuid)
  - schoolName (string)
  - grade (int)
  - classNum (int)
  - studentNumber (int)
  - heightCm (int nullable)
  - weightKg (int nullable)
  - allergies (string[] or JSON)
  - createdAt, updatedAt
- 유니크 제약: (schoolName, grade, classNum, studentNumber) 조합 unique
- (선택) Meal (오늘 급식 placeholder 저장용)
  - id, date, schoolName, menuText, caloriesKcal(optional), createdAt

E. 보안/구조(이번 단계에서 꼭)
- 서버/클라 분리 원칙 준수(민감 데이터는 API 통해서만)
- 입력값 검증(Zod) + 에러 응답 형식 통일
- 세션 없으면 /dashboard 접근 시 /login으로 redirect
- env 관리:
  - DATABASE_URL
  - SESSION_SECRET (또는 JWT_SECRET)
- 향후 확장 포인트를 주석으로 최소 표시:
  - NEIS 연동은 /api/meals/today 내부에서 교체 가능
  - AI 기능은 별도 service 레이어 추가 가능
  - 포인트/피드백/사진 업로드는 테이블 확장으로 연결 가능

[구현 방식 지시]
- Next.js App Router 기준으로 작성.
- server action을 써도 되지만, 이번 단계는 이해 쉬운 “API route + fetch” 방식으로 통일해도 됨.
- UI 컴포넌트는 과도한 라이브러리 없이, 작은 components 폴더에 Button/Input/Card 정도만 만들어 재사용.
- 에러 처리 UX는 미니멀: 폼 아래에 한 줄로 빨간 텍스트 정도.
